# ヤマトマスタパック API実装方針

## マスタファイル仕様

### YMSTPOST.DAT（郵便番号→仕分コード）

- レコード長: 50バイト（固定長、Shift_JIS、CRLF改行）
- 件数: 約142,000件
- サイズ: 約7MB

| 位置 | 長さ | 内容 |
|------|------|------|
| 1 | 1 | 1固定（無視） |
| 2-8 | 7 | 郵便番号 |
| 9-11 | 3 | スペース |
| 12-18 | 7 | 仕分コード1（※2024年版以降はこちらを使用） |
| 19-25 | 7 | 仕分コード2 |
| 26-42 | 17 | 日付・区分等（詳細不明） |
| 43-50 | 8 | 更新日（YYYYMMDD） |

**仕分コード（7桁）の構造:**
- 先頭3桁 = ベースNo（発送拠点番号）
- 先頭5桁 = YMSTTIME検索用の仕分コード
- 全7桁 = 完全な仕分コード

### YMSTTIME.DAT（リードタイム判定）

- レコード長: 40バイト（固定長、CRLF改行）
- 件数: 約473,000件
- サイズ: 約19MB
- 発ベース: 105拠点 × 着仕分コード: 4,508種 の組み合わせ

| 位置 | 長さ | 内容 |
|------|------|------|
| 1-3 | 3 | 発ベースNo |
| 4-8 | 5 | 着仕分コード（先頭5桁） |
| 9-10 | 2 | スペース |
| 11-12 | 2 | 最短配達日数（02=翌日, 03=翌々日... 11=配達不可） |
| 13-14 | 2 | 最短受取開始時間帯（08=午前, 14, 16, 18, 99=指定不可） |
| 15-16 | 2 | 最短受取終了時間帯（通常20=20時まで, 99=指定不可） |
| 17-32 | 16 | サービスレベル詳細（用途不明） |
| 33-40 | 8 | 更新日（YYYYMMDD） |

### その他のマスタ（参考）

| ファイル | サイズ | 用途 |
|---------|--------|------|
| YMSTAREA.DAT | 1.6KB | ベースと地域の対応（14件） |
| YMSTJIS5.DAT | 448KB | 行政コード→仕分コード（1,892件、Shift_JIS） |
| YMSTMTRX.DAT | 35KB | 発ベース→着ベース別の配達日数マトリクス（106件） |

## API① 着郵便番号から仕分コード検索

### エンドポイント

```
GET /yamato/postcode/{zip}
```

### リクエスト

| パラメータ | 型 | 説明 |
|-----------|-----|------|
| zip | string | 郵便番号7桁（ハイフンなし） |

### レスポンス例

```json
{
   "zip": "1860002",
   "sort_code": "0331502",
   "base_no": "033",
   "updated_at": "20231205"
}
```

### 実装方針

1.
起動時にYMSTPOST.DATを読み込み、郵便番号をキーとした連想配列（ハッシュマップ）に格納
2. リクエスト時はO(1)で検索
3. メモリ使用量: 約30-50MB（7MB × 連想配列オーバーヘッド）

## API② 宅急便の最短お届け日検索

### エンドポイント

```
GET /yamato/leadtime?from_zip={発zip}&to_zip={着zip}&ship_date={出荷予定日}
```

### リクエスト

| パラメータ | 型 | 説明 |
|-----------|-----|------|
| from_zip | string | 発送元郵便番号7桁 |
| to_zip | string | 発送先郵便番号7桁 |
| ship_date | string | 出荷予定日（YYYY-MM-DD） |

### レスポンス例

```json
{
  "from_zip": "1860002",
  "to_zip": "9040004",
  "ship_date": "2026-06-20",
  "from_base_no": "033",
  "to_sort_code": "09851",
  "delivery_days": 3,
  "earliest_delivery_date": "2026-06-22",
  "earliest_time_from": "08",
  "earliest_time_to": "20",
  "deliverable": true
}
```

- `delivery_days`: マスタ上の最短配達日数（2=翌日着、3=翌々日着）
- `earliest_delivery_date`: ship_date + (delivery_days - 1)
で算出した最短届日

### 検索ロジック

```
1. 発送元zip → YMSTPOST → 仕分コード1の先頭3桁 = 発ベースNo
2. 発送先zip → YMSTPOST → 仕分コード1の先頭5桁 = 着仕分コード
3. 検索キー = 発ベースNo(3桁) + 着仕分コード(5桁)
4. YMSTTIME → 検索キーで該当レコード取得
5. pos11-12 = 最短配達日数, pos13-14 = 受取開始時間, pos15-16 = 受取終了時間
```

### 実装方針

1. 起動時にYMSTPOST.DAT + YMSTTIME.DATを読み込み
2. YMSTPOST: 郵便番号→仕分コード1の連想配列
3. YMSTTIME: 発ベースNo+着仕分コード(8桁)→レコードの連想配列
4. メモリ使用量: 合計約80-120MB
5. Lambda割り当て: 256MB〜512MBで十分

## 技術選定と方針

### 言語: TypeScript（Node.js）

**Why:**
- CDKもアプリも同一言語で完結（リポジトリ構成がシンプル）
- 固定長パースは `Buffer` + `iconv-lite`（Shift_JIS→UTF-8）で十分
- 142k件+473k件のMapはV8エンジンで余裕
- Lambda Web Adapter との相性が良い（Fastify等でHTTPサーバを立てるだけ）

### フレームワーク: Fastify

**Why:**
- Express より高速（JSON serialize が速い）
- スキーマバリデーション内蔵（zip 7桁チェック等）
- Web Adapter は `localhost:8080` を待ち受けるだけなので軽量HTTPサーバで十分

### デプロイ: AWS CDK（TypeScript）

**Why:**
- アプリと同一言語 → 型の共有やモノレポ運用が容易
- Lambda Function URL + IAM認証の設定が数行で書ける
- ECRリポジトリ・CodeBuildも同一スタックで管理可能

### 認証: Lambda Function URL（IAM認証）

**Why:**
- API Gatewayを挟まない → コスト削減＆レイテンシ低減
- IAM認証 → 呼び出し元にAWSクレデンシャルがあれば即利用可能
- EC/業務システム側はSigV4署名で呼び出し（SDK利用で簡単）

## インフラ構成案（Lambda Web Adapter）

```
[CodeBuild 週1定期ビルド]
     ↓ DATファイルDL & イメージビルド
[ECR]
     ↓
[Lambda (コンテナイメージ, 512MB)]
     ↓
[Lambda Function URL (IAM認証)]
```

- DATファイルはDockerイメージに焼き付け（ビルド時にDL）
- 起動時に1回だけ読み込み→連想配列化
- Web Adapterでプロセス常駐→2回目以降のリクエストは即応答
- 全データ合計27MB、連想配列化後も512MB以内に十分収まる
